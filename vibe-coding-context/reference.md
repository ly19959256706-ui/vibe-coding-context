## 共同规则

### 先分级再执行

每个新任务都先完成 A–F checklist。按 `heavy → medium → light → tiny` 的优先级选择唯一档位，不凭感觉混用工作流。

### 渐进读取项目上下文

当 B 命中时：

1. 读取项目已有 `AGENTS.md`。
2. 读取 `docs/context-index.md` 或项目已有的上下文索引。
3. 若没有索引，先查看附近文档文件名；只有在确实能减少后续误读时才创建轻量索引。
4. 只读取当前步骤需要的需求、业务、架构、接口、UI 或决策文档。
5. 保留原始文档路径；索引只是导航，不是事实来源。

不得为了适配本 skill 复制或重命名用户文档。长期有效的产品、架构和设计决策应写回项目现有决策文档，而不是永久堆在任务记忆里。

### 单一写入位置

| 活动档位 | 允许写入 | 禁止写入 |
|---|---|---|
| tiny | 任务交付物 | `ai/`、`.planning/` |
| light | `ai/task-brief.md` | `ai/progress.md`、`ai/findings.md`、`.planning/` |
| medium | `ai/` 三文件 | `.planning/` |
| heavy | `.planning/<任务>/` 三文件 | `ai/` 任务记忆 |

## tiny：零规划文件

### 适用场景

- 修改一处文案。
- 修正单文件拼写或样式。
- 调整一个配置值且不需要跨会话恢复。
- 回答一次性问题。

### 工作步骤

1. 确认 A–F 全部未命中。
2. 直接检查相关文件或事实。
3. 完成修改或回答。
4. 运行与风险相称的最小验证。
5. 交付结果。

### 完成条件

没有额外规划门禁。验证与交付完成即可结束。

## light：单一任务简报

### 适用场景

命中 1–2 项且全部属于 A/B/C。例如跨三个文件的小功能、需要读取一份现有 UI 规范的局部调整，或预计稍多于五次工具调用但没有验收门禁和跨会话要求。

### 文件

只使用 `ai/task-brief.md`。从 `templates/ai/task-brief.md` 创建；若项目已有等价活动任务简报，直接复用，不创建重复文件。

### 字段

- 分级结果：记录命中的 A–F 项和当前档位。
- 目标：一句话说明预期结果。
- 范围：明确包含与不包含的工作。
- 验收标准：可保持简短；light 不依赖自动门禁。
- 当前步骤：写下一项可执行动作。
- 状态：使用 `进行中`、`待用户确认` 或 `已完成`。
- 开放问题：只保留真正影响方向的问题。

### 更新时机

只在目标、范围、当前步骤或阻塞发生实质变化时更新。不要在每次编辑后写进度日志。

### 完成条件

完成修改和必要验证，将状态设为 `待用户确认`，交付结果。用户确认后设为 `已完成`。

## medium：轻量三文件

### 适用场景

命中 E，或在未命中 D/F 的前提下总命中数为 3–4。典型场景包括带明确验收标准的功能切片、同时涉及需求/UI/实现/测试的中型功能，以及需要自然里程碑记录但不必建立重型阶段证据链的任务。

### 文件与职责

| 文件 | 字段 | 更新时机 |
|---|---|---|
| `ai/task-brief.md` | 分级、目标、范围、验收、当前步骤、状态、开放问题 | 目标、范围、验收或当前步骤变化时 |
| `ai/progress.md` | 当前状态、已完成、验证记录、下一步、交接摘要、近期历史 | 功能切片、测试、下一步或交接时 |
| `ai/findings.md` | 项目事实、临时决策、已解决问题、参考资料 | 发现可复用事实、做出临时决策或解决易复发问题时 |

### 里程碑规则

- 一个可验证的功能切片完成后更新 progress。
- 测试运行后记录方法、结果和结论。
- findings 只保留能帮助后续任务的信息，不复制大段原始文档。
- 临时决策稳定后提升到项目决策文档，并在 findings 保留指针。
- 会话即将结束且任务未完成时补充交接摘要和明确下一步。

### 恢复

依次读取 `ai/task-brief.md`、`ai/progress.md`、`ai/findings.md`，再按上下文索引读取下一步所需的真实项目文档。先简短说明目标、当前状态和下一步，然后继续工作。

### 完成条件

逐项核对 task brief 中的验收标准。所有验收项均勾选后，把验证结果写入 progress，并将状态设为 `已完成`。任何验收项未满足时不得宣称完成。

## heavy：阶段计划与证据链

### 适用场景

命中 D 或 F，或总命中数 ≥5。典型场景包括研究、审计、迁移、复杂排错、跨阶段开发、多天任务和需要可靠交接的工作。

### 目录约定

始终使用：

```text
<基准目录>/.planning/<任务名>/
├── plan.md
├── progress.md
└── findings.md
```

项目模式以识别到的项目根为基准目录；standalone 模式以当前工作目录为基准目录。中文任务名会保留中文和数字，其他连续符号转换为单个连字符。

显式运行：

```powershell
& '<skill目录>\scripts\init-planning.ps1' `
  -TaskName '<任务名称>' `
  -WorkingDirectory (Get-Location) `
  -Mode Auto
```

读取输出的 `PLANNING_DIR`，后续所有 heavy 记录只使用该目录。

### plan.md

- 目标：说明最终结果。
- 范围：列出包含与不包含内容。
- 验收标准：使用可勾选条目。
- 当前阶段：只允许一个 `in_progress`。
- 阶段：每个阶段使用 `pending`、`in_progress` 或 `complete`。
- 关键决策：记录会影响实现或结论的选择及理由。
- 阻塞与升级：只记录阻塞、复发或影响正确性的错误。
- 完成门禁：验收、测试、阶段三项必须同时通过。

重大决策和阶段切换前重读 plan。不得仅因为停止工作就把阶段标为 complete。

### progress.md

- 按阶段记录已执行动作和产物。
- 记录测试方法、预期、实际结果和结论。
- 在长时间等待、会话结束或交接前写明当前状态和下一步。
- 只记录有恢复价值的过程，不逐条复制工具调用。

### findings.md

重要结论使用以下证据结构：

| 事实 | 推论 | 来源 | 日期 | 可信度 |
|---|---|---|---|---|
| 可直接验证的观察 | 事实意味着什么 | 文件、命令、数据集或网址 | 观察或发布日期 | 高/中/低 |

事实和推论必须分开。查看图片或 PDF 后立即把关键视觉信息转成文字；完成一个证据块、切换来源/环境/方案或即将交接时保存 findings。

### 恢复

1. 若用户明确任务名，优先读取对应 `.planning/<任务名>/`。
2. 否则枚举当前项目或工作目录 `.planning/*/plan.md`。
3. 选择最近修改且仍包含 `pending` 或 `in_progress` 阶段的任务。
4. 若没有活动任务，重新分级当前请求，不自动恢复已完成任务。
5. 依次以 UTF-8 读取 plan、progress、findings。
6. 确认目标、当前阶段、已知证据、已完成动作和下一步，再继续工作。

本流程不读取聊天会话日志，不依赖宿主 hooks。

### 完成门禁

只有以下三项同时满足，heavy 才能完成：

1. plan 的所有验收标准均已勾选。
2. progress 至少有一条与验收相关的测试记录，且所有测试结论为通过。
3. plan 的所有阶段状态均为 `complete`。

任何一项不满足时继续执行安全且在范围内的下一步；确实需要用户决策时说明阻塞。

## 档位升级

### tiny 升级

从升级时开始创建 light 或 medium 所需文件，不补写此前已完成的动作。

### light 升级到 medium

保留 task brief，创建 progress 和 findings。三份文件仍属于同一个 `ai/` 记录域。

### 升级到 heavy

在原 task brief 顶部写入：

```text
> 已升级到 heavy，后续记录见 .planning/<任务名>/
```

随后冻结所有 `ai/` 文件，初始化 `.planning/<任务名>/`。不得迁移、复制或同步旧内容；必要背景只通过原文件路径引用。

## 完成检查

| 档位 | 检查方式 |
|---|---|
| tiny | 无规划检查 |
| light | 交付并由用户确认 |
| medium | task brief 的验收标准全部满足 |
| heavy | 验收全勾选、测试全通过、阶段全为 complete |
